2. HTML Form

You need to create a registration form for the new site you are building. When a user submits the form (clicking the submit button), the resulting HTTP request should include at least the following:

POST /wsd/html_form_check HTTP/1.1
Host: 127.0.0.1

Think about this for a minute. What doest that POST mean? What about the Host: header? How are they related to submitting an HTML form? Make sure you use this exact address and path combination in the file you submit!

An example of the parameters in the body of the request, as well as their explanations:

Name / IDExample ValueExplanation
emailtest@example.comUser's email address
passwdsalasanaPassword
passwd2salasanaRepeat password
langdaSelection of user's preferred language: the application targets the nordic region, so the user can choose one from Swedish (submitted value sv), Finnish (fi), Norwegian (no), Danish (da), and Icelandic (is).
termsonCheckbox to accept site's terms and conditions

The example values above are just examples - Do not pre-populate the fields with data or pre-check checkboxes.

Important! Set the name and id attributes for the input- and select-elements as given in the table above. Otherwise the submission system cannot find the elements.

Example of a form:

Since you consider accessibility an important part of a user experience, you want all the form fields to have label elements. The easiest way to test that the labels are working is to make sure that clicking the label moves the focus to the input element.

Where to start

Create a new file exercises/02_html_css/form.html. In addition to the form, the file should contain required HTML elements (html, head, title, body), that is, it should be valid HTML.

If you did not already start using a text editor that properly supports web development now would be the recommended time to do so. The time invested in development environment will pay back in an exercise or two. Some recommendations: Atom-editor, in preferences one can add packages, such as: linter, linter-htmlhint, linter-csslint, tag.

Testing your form

When developing your form, we suggest that you test your form by submitting it to a service called RequestBin. Create your own RequestBin at http://requestbin.net/ by clicking "Create a RequestBin" there. You will receive a url looking something like http://requestbin.net/r/yl2n61yl (the yl2n61yl part is different). You should use the given address to test your form instead of the required one having 127.0.0.1 in it but just remember to switch it back before you submit your code. Then fill out the form and submit it and visit the same url with "?inspect" (eg. http://requestbin.net/r/yl2n61yl?inspect) to inspect the data that got sent to the the RequestBin. Do not use real passwords when testing your form!

3. CSS Layout

In this exercise, your task is to create a simple page layout with CSS.

The page to style is exercises/02_html_css/html-layout.html. The HTML consist of div elements with descriptive id attributes for the different parts of the page:

header
The header of the page. This also contains the navigation links that should be made into a horizontal navigation bar.
main
The main part of the page where the content element and an aside providing resources related to the content are.
footer
The footer of the page.

You should write CSS so that the positioning of the elements looks like the figure below. You do not have to match the styling of the content (that is, the borders, heights, etc).

Note: Use floats to achieve the layout. Do not use absolute/fixed positioning. Also, do not use pixel based widths; you can make the aside element, for example, 20% wide and the content element 75% wide.

Where to start

Create a new file exercises/02_html_css/layout.css. Open the exercises/02_html_css/html-layout.html in browser. Note, that the grading is done using the original HTML file. Do do not make your answer depend on changes to the HTML.

Hints if you have problems with the grader

The grader is quite picky, so here are some things to check if you're not getting the grade you would assume:

  • Do not use pixel-based widths. The grader runs the tests with a low resolution, so the components might not fit to the screen.
  • Do not use big pixel-based margins, padding, or borders to change your layout. Again, low resolution on grader...
  • Using CSS overflow directives can cause problems in the grader browser environment.
